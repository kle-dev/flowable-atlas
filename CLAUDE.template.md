## 1. What Flowable is (the mental model an LLM usually gets wrong)

Flowable is a Java process-automation platform. A "solution project" is custom Java + models
that run **on top of** the Flowable engines — you are extending a platform, not building from scratch.

**The four products:**
| Product | Role |
|---|---|
| **Work** | The **runtime *and* end-user frontend** (React UI). Executes deployed definitions, renders forms, hosts tasks/cases. Custom Java + REST controllers live here. |
| **Design** | The **visual modeler**. Models are editable JSON; users design BPMN/CMMN/DMN/forms/apps here. |
| **Control / Hub** | Admin/monitoring consoles for Work clusters (Hub = newer React version). |
| **Engage** | Conversational/omnichannel layer (often not used). |

**The single most important concept — Models vs Definitions:**

```
DESIGN (Models, editable JSON in ACT_DE_MODEL)        WORK (Definitions, deployed & IMMUTABLE)
  App (package model)  ── publish (export ZIP) ──►   Deployment  (POST /app-repository/deployments)
    ├─ BPMN model      ─────────────────────────►      ProcessDefinition   (ACT_RE_PROCDEF)
    ├─ CMMN model      ─────────────────────────►      CaseDefinition      (ACT_RE_CASE_DEF)
    ├─ DMN model       ─────────────────────────►      DecisionDefinition
    ├─ Form model      ─────────────────────────►      FormDefinition
    └─ Data object / service / query / ...       ►      (deployed platform definitions)
```

- Models are **mutable** (design time). Definitions are **immutable, versioned** (runtime, in the DB).
- Everything is referenced by **key** (e.g. a process `key`, form `key`, decision `key`). Cross-references
  between models are by key — that is what Atlas resolves.
- Runtime state lives in `ACT_RU_*`, history in `ACT_HI_*`, deployed definitions in `ACT_RE_*`,
  identity (users/groups) in `ACT_ID_*`, design models in `ACT_DE_*`.

**In a solution project, the models in this repo are *exported from Design*** — the `.app`/`.zip` and
model files under `src/main/resources` are **exported build artifacts**, not the editing surface. The
Java app is built **together with** the bundled model and deploys it to Work on startup. To *change* a
model, the canonical path is Flowable **Design**, then re-export into the repo.

**Engines:** BPMN (processes), CMMN (cases), DMN (decisions), Form, Content, IDM (users/groups),
plus platform engines (data objects, actions, agents, indexing/Elasticsearch, etc.).

## 2. How custom code attaches to models (the extension points)

A solution project's Java/frontend hooks into models at well-defined points. When you implement a
feature, you almost always touch one or more of these:

- **JavaDelegate / service tasks** — a BPMN/CMMN service task runs Java via
  `flowable:class="com.acme.MyDelegate"` or `flowable:delegateExpression="${myBean}"`
  (a Spring `@Component`/`@Service`). The bean is the unit of business logic.
- **Expressions** — `${bean.method(args)}` (backend, JUEL) anywhere in models (conditions,
  listeners, field injections). Frontend bindings use `{{ ... }}` in forms/pages.
- **Listeners** — `ExecutionListener`, `TaskListener`, `PlanItemInstanceLifecycleListener`,
  `CaseInstanceLifecycleListener`, `FlowableEventListener` for reacting to lifecycle events.
- **REST controllers** — `@RestController` classes expose endpoints that the Work frontend (forms,
  data tables, buttons) calls. Forms reference them via `url`/query bindings.
- **Bots** (`BotService`) — invoked by **Actions** (`.action` models, referenced by `botKey`).
- **Service-registry data objects** — `.data` models backed by a `.service` (REST or DB). DB-backed
  ones map to tables defined in **Liquibase** changelogs (`referencedLiquibaseModelKey`, `tableName`).
- **Forms** — bind fields to **variables**; outcomes drive flow; can call REST for options/data tables.
- **Queries** (`.query`) — index queries (over `tasks`/`case-instances`/…) used by dashboards/lists;
  often gate results by **user group** (`currentGroups?seq_contains(...)`).
- **Variables** — process/case variables: set in Java (`execution.setVariable("x", …)`), via
  init-var mappings (`flowable:variableMapping`), in/out parameters, sequences, or read in expressions.
- **Access control** — candidate (starter) groups on processes/cases, candidate groups/assignees on
  tasks, app/page `accessPermissions`, data-object identity links, security policies.

## 3. How models, code and deployment fit together (important — easy to get wrong)

The model and the Java are developed separately and **built together**, then promoted through
environments. The agent does **not** drive Design.

- **Models are authored in Design, not here.** A modeler builds BPMN/CMMN/forms/etc. in Flowable
  **Design**, then **exports/publishes the app into this repo**. The repo's model files are exported
  artifacts — **don't hand-edit the deployed `.zip` as if you own it**; model changes normally go back
  through Design and are re-exported.
- **Build = your Java + the bundled model, together.** The Maven build packages the custom Java **and**
  the exported app; on deploy/startup the app **auto-deploys** its definitions to Work.
- **Deploy via the built artifact, environment by environment** (dev → test → **prod**, via CI/CD). You
  typically do **not** publish from Design straight to Production — Design-publish is a dev-time
  convenience; production gets the built-and-deployed app.

**Your lane as an agent:**
1. Implement/adjust the **custom Java** (delegate/bean/listener/REST controller/bot) and wire it to
   **match** the model (`delegateExpression`/`flowable:class`/listener config — names must line up).
2. **Read** the models to understand the wiring (use Atlas).
3. If a feature needs a **model change** (new task/form/variable/decision), **say so explicitly and
   describe it** — it's made in Design and re-exported, unless this project's convention is to edit the
   model files directly (check existing commits/patterns first).
4. Add **Liquibase** + `.service`/`.data` if a custom table is needed; add/adjust automated tests.

Always mirror an existing similar case — find it via Atlas.

## 5. Rules for the agent

- **Understand before coding:** use `APP_SUMMARY.md` → `graph.json` → source. State which existing
  process/case/form/bean your feature builds on.
- **Verify, don't hallucinate:** Flowable APIs differ across versions. Confirm method/class names
  against the actual dependencies and the official docs (https://documentation.flowable.com) — do
  not invent engine APIs.
- **Match model ↔ code:** if you add a `delegateExpression`/`flowable:class` in a model, the bean/class
  must exist (and vice-versa). Atlas's "Unresolved references" section shows mismatches.
- **Keys are contracts:** processes/cases/forms/decisions are referenced by key across models and from
  Java (`processDefinitionKey("…")` etc.) and the frontend. Renaming a key breaks references — check
  Atlas for who references it (both directions) first.
- **Respect access/security:** candidate groups, app/page permissions and security policies are part of
  the feature, not an afterthought.
- **Minimal, consistent changes:** mirror existing patterns; touch only what's necessary.

**Common Flowable pitfalls (the knowledge gap):** don't invent engine APIs (verify against deps/docs);
don't hand-edit the exported app `.zip` (model changes go via Design); mind variable scope
(`setVariable` vs `setVariableLocal`); never rename a definition `key` without checking who references it
(both directions); don't forget candidate groups / access on new tasks & pages.
